  - **In-scope**: CI/CD, 테스트, Observability, Persistent queue (SAQ), DB 마이그레이션 (SQLModel), Webhook security, DI 리팩터링, aiogram 기본 연동

  - **30d**: CI/CD + Test ≥60% + structlog + DI 리팩터

## B. Context & Requirements (PRD-lite)

### B1. 문제정의
- **현재 문제**: logi_orchestrator는 잘 설계된 상태 머신과 API 스켈레톤을 갖추고 있으나, 모든 외부 연동이 stub이고 인프라(queue/DB/로깅/테스트/CI)가 전무. 프로세스 재시작 시 모든 Job 데이터가 유실되며, 테스트 3개로 회귀 방지 불가능.
- **왜 지금 해야 하는가**: HVDC 프로젝트의 Telegram 기반 멀티에이전트 오케스트레이션을 실제 운영하려면, stub에서 벗어나 데이터 영속성·관측성·품질 게이트가 확보되어야 한다. 이 인프라 없이 Claude/Codex 실연동을 진행하면 디버깅 불가 + 데이터 유실 리스크가 극대화됨.

### B2. 사용자/페르소나
- **Primary**: Cha (SCM/Logistics Manager) — Telegram에서 `/project.start` 명령으로 작업 시작, 승인, 상태 확인
- **Secondary**: DevOps/개발자 — CI/CD 모니터링, 장애 대응, 코드 리뷰

### B3. 사용자 시나리오/유저스토리
- **Story 1**: Cha가 Telegram에서 `/project.start repo=macho715/logi_hvdc_dash goal="ETA KPI 탭 추가"`를 보내면, Orchestrator가 Job을 생성하고 Plan 생성을 시작한다. 서버가 재시작되어도 Job 상태가 보존된다.
- **Story 2**: DevOps가 structured JSON 로그에서 `request_id`로 특정 Job의 전체 요청 흐름을 추적하여, webhook 수신 → plan 생성 → approve → merge까지의 경로를 5분 내에 파악한다.
- **Story 3**: 개발자가 PR을 올리면 GitHub Actions에서 ruff+mypy+pytest가 자동 실행되어, 코드 품질 회귀를 사전 차단한다.

### B4. 요구사항

**Functional requirements**:
- Persistent job queue (프로세스 재시작 후 Job resume)
- ACID-compliant job state storage
- Structured JSON logging with request_id correlation
- Telegram webhook HMAC signature verification
- Idempotency key 기반 중복 요청 방지
- CI pipeline (lint + type check + test) on every PR

**Non-functional requirements (SLO)**:
- Webhook → Job 생성: p95 < 500ms
- SAQ dequeue latency: p99 < 10ms
- DB query: p99 < 20ms
- CI pipeline: < 3min
- Uptime target: 99.5% (dev/staging)

**Constraints**:
- Python 3.11+ only
- No Docker required for dev (SQLite + fakeredis)
- GitHub Actions free tier (2000 min/month)
- Redis required for SAQ in production
- 코드 자동 변경/배포 금지 — 문서+PR만

## D. System Architecture (Components & Boundaries)

### D1. High-level Components

```
[Telegram Group]
    │ webhook (HMAC verified)
    ▼
[FastAPI API Layer]
    ├─ Middleware: logging (structlog+request_id), metrics (prometheus), rate_limit
    ├─ Routes: telegram, jobs, health, metrics
    ├─ Services: command_parser, job_service, acl, audit, state_machine
    │      │ Depends() injection
    │      ▼
    ├─ Infra: SQLJobStore (SQLModel+Alembic), SAQ Queue, AuditService
    │      │
    │      ├──[PostgreSQL/SQLite]  (Job records, Audit events)
    │      └──[Redis]              (SAQ task queue)
    │
    └─ Workers: SAQ Worker
           ├─ PlanHandler (stub → future Claude API)
           ├─ ExecHandler (stub → future Codex CLI)
           ├─ TestHandler (stub → future pytest runner)
           ├─ ReviewHandler (stub → future Claude Review)
           └─ MergeHandler (stub → future Git merge)
```

### D2. Data Flow

**Request flow**:
1. Telegram → `POST /webhooks/telegram` (HMAC verify → rate limit → idempotency check)
2. Command parser → CommandEnvelope
3. JobService.create_job() → SQLJobStore.create() + SAQ.enqueue(PLAN task)
4. Response → WebhookAck JSON

**Background jobs**:
1. SAQ Worker dequeue → task dispatch by JobTaskType
2. Handler: read job from DB → execute (stub/real) → write artifacts → transition state → save DB → audit log
3. If next task needed → SAQ.enqueue() (chaining)

**Caching strategy**:
- ACL config: in-memory cache with TTL 5min (reload on change)
- Job record: no cache (always DB read for consistency)
- Redis: SAQ queue only (no application-level cache yet)

### D3. Boundary Rules
- **Layering**: Routes → Services → Infra (단방향, 역참조 금지)
- **Allowed imports**: routes는 services+schemas만, services는 infra+schemas+core만, infra는 core+schemas만
- **DI 원칙**: 모든 Service/Store는 FastAPI Depends()로 주입. 전역 인스턴스 금지.
- **변경 전파 표면 최소화**: Store/Queue는 Protocol(ABC) 정의 → 구현체 교체 가능

## E. Data Model & API Contract

### E1. Data Model (Entity)

| Entity | Fields (key) | Source of truth | Validation | Notes |
|---|---|---|---|---|
| JobRecord | job_id (PK), repo, base_branch, goal, acceptance_criteria, mode, priority, requested_by, approved_by, state (JobState enum), previous_state, resume_target_state, current_phase, retry_count (JSON), trace_id, chat_id, created_at, updated_at, run_dir, artifacts (JSON list), notes (JSON dict) | PostgreSQL / SQLite | Pydantic v2 + SQLModel | state는 JobState enum으로 제약 |
| AuditEvent | id (PK auto), ts, job_id (FK), from_state, to_state, event, actor, trace_id, detail (JSON) | PostgreSQL / SQLite | Pydantic v2 | append-only, 삭제 금지 |
| JobTask (Queue) | job_id, task_type (enum), actor, trace_id, payload (JSON) | Redis (SAQ) | Pydantic v2 | transient, 처리 후 Redis에서 제거 |
| ACLConfig | requesters, approvers, operators, repos | YAML file (config/acl.yaml) | PyYAML + Pydantic | in-memory cache, TTL 5min |

### E2. API Contract

| Endpoint | Method | Auth | Request | Response | Error codes |
|---|---|---|---|---|---|
| `/webhooks/telegram` | POST | HMAC secret token | TelegramUpdate JSON | WebhookAck | 401 (bad signature), 429 (rate limit), 400 (validation) |
| `/jobs/{job_id}` | GET | - | path: job_id | JobRecord JSON | 404 |
| `/jobs/{job_id}/approve-plan` | POST | X-Actor header | path: job_id | WebhookAck | 400, 403, 404 |
| `/jobs/{job_id}/approve-merge` | POST | X-Actor header | path: job_id | WebhookAck | 400, 403, 404 |
| `/jobs/{job_id}/pause` | POST | X-Actor header | path: job_id, body: reason | WebhookAck | 400, 403, 404 |
| `/jobs/{job_id}/resume` | POST | X-Actor header | path: job_id | WebhookAck | 400, 403, 404 |
| `/jobs/{job_id}/abort` | POST | X-Actor header | path: job_id, body: reason | WebhookAck | 400, 403, 404 |
| `/jobs/{job_id}/artifacts` | GET | - | path: job_id | {artifacts: [...]} | 404 |
| `/health` | GET | - | - | {status, checks} | 503 (unhealthy) |
| `/metrics` | GET | - | - | Prometheus text | - |

### E3. AuthN/AuthZ
- **RBAC 모델**: requester / approver / operator (config/acl.yaml 기반)
- **Token/session 전략**: Telegram webhook → HMAC secret token header 검증. API 직접 호출 → X-Actor header (dev), 향후 JWT 전환 고려.
- **Audit log 요구**: 모든 state transition + approval은 AuditEvent 테이블에 기록. append-only, 삭제 금지.

## I. Observability & Operations

### I1. Logging/Tracing/Metrics

**Logs (structlog)**:
- Format: JSON (machine) / colored (dev console)
- Fields: `timestamp`, `level`, `request_id`, `job_id`, `event`, `actor`, `duration_ms`
- Rotation: stdout (container) / file with rotation (standalone)

**Metrics (Prometheus)**:
- `http_request_duration_seconds` (histogram, by method/path/status)
- `http_requests_total` (counter, by method/path/status)
- `job_state_transitions_total` (counter, by from_state/to_state/event)
- `saq_queue_length` (gauge)
- `saq_task_duration_seconds` (histogram, by task_type)

**Tracing (OpenTelemetry)**:
- Auto-instrumentation: FastAPI, httpx, SQLAlchemy
- Custom spans: `job_service.create_job`, `worker.handle_plan`, state transitions
- Exporter: OTLP (Jaeger/Tempo) or NoOp (flag off)

### I2. Alerting & On-call
- **Alert rules** (staging 이후):
  - p95 latency > 1s for 5min → warn
  - Error rate > 5% for 3min → critical
  - SAQ queue length > 50 for 10min → warn
  - Health check fail 3x → critical
- **Escalation**: Telegram Bot → 관리자 DM (향후)
- **SLO breach 대응**: dashboard 확인 → 원인 분류 (app/db/redis/external) → runbook 실행

### I3. Runbooks

**Deploy**:
1. `git pull` → `pip install -r requirements.txt`
2. `alembic upgrade head`
3. `uvicorn app.main:app --host 0.0.0.0 --port 8000`
4. Health check: `curl /health`

**Rollback**:
1. `git checkout <previous-tag>`
2. `alembic downgrade -1`
3. Feature flag 복원: `STORE_BACKEND=file`, `QUEUE_BACKEND=memory`
4. Restart uvicorn

**Incident response**:
1. 증상 확인 (로그/메트릭)
2. request_id로 trace 추적
3. 영향 범위 파악 (job_id 목록)
4. 원인 분류 → runbook 실행
5. Post-mortem 작성

## J. Error Handling & Recovery

### J1. Error Taxonomy

| Category | HTTP | Examples | Action |
|---|---|---|---|
| 사용자 입력 오류 | 400 | 잘못된 명령, 누락 인자, 잘못된 job_id 형식 | 명확한 에러 메시지 + 재입력 안내 |
| 인증/권한 오류 | 401/403 | HMAC 실패, ACL 미허용 | 에러 로그 + 거부 응답 |
| 리소스 미존재 | 404 | job_id 없음 | 에러 응답 |
| 상태 전이 오류 | 409 | 잘못된 상태에서 approve 시도 | 현재 상태 + 허용 action 안내 |
| Rate limit | 429 | 과다 요청 | Retry-After 헤더 |
| 서버 오류 | 500 | DB 연결 실패, Redis 장애, 코드 버그 | 에러 로그 + generic 메시지 |
| 외부 의존성 오류 | 502/503 | Telegram API 장애, Claude/Codex 타임아웃 | retry → circuit breaker |

### J2. Retry/Timeout/Idempotency

- **Timeout 정책**:
  - Webhook 처리: 10s hard limit
  - SAQ task: 300s (5min) per task, configurable
  - DB query: 5s
  - External API (향후): 30s

- **Retry 정책**:
  - SAQ task: max 2 retries, exponential backoff (10s, 30s)
  - DB connection: 3 retries with 1s interval
  - Telegram sendMessage: 3 retries with backoff

- **Idempotency**:
  - Telegram update_id dedup (Redis SET with TTL 30s)
  - approve.plan/approve.merge: 이미 승인된 경우 NO-OP 200 응답
  - trace_id 기반 30s 내 동일 요청 거부

- **Circuit breaker (향후)**:
  - Claude API / Codex CLI 연속 3회 실패 → open (30s) → half-open (1 probe) → close
  - Telegram sendMessage 연속 5회 실패 → queue에 적재, 3min 후 재시도

### J3. UX Error Messaging (Telegram)

```
# 입력 오류
NACK
reason=INVALID_ARGUMENT
missing=goal
hint=goal="작업 목표를 입력하세요"

# 상태 오류
NACK
reason=INVALID_STATE
current=EXEC_RUNNING
allowed=pause, abort

# 서버 오류
ERROR
reason=INTERNAL_ERROR
trace_id=tg-20260322-0042
action=잠시 후 재시도하거나 관리자에게 문의하세요
```
